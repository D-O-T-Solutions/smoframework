# RFC 0005 — Execution DAG

## Status
ACCEPTED — incorporated into SPEC.md §XII.

## Problem
How to represent a multi-step intent so that execution is deterministic, auditable, and parallelizable?

## Decisions
1. **DAG after compilation.** The compiler transforms an intent into a directed acyclic graph of tasks.
2. **DAG is immutable after compile.** No runtime component may modify it.
3. **Explicit dependency edges.** A task lists its dependencies by task_id.
4. **DAG-aware scheduler.** The scheduler is not a FIFO queue; it traverses the graph respecting dependencies.

## Rationale
- Immutability guarantees that every node executes the same plan — critical for audit and replay.
- Explicit dependencies enable natural parallelism (independent tasks run concurrently).
- DAG structure maps directly to most incident-response and orchestration workflows.

## Consequences
- The compiler must reject cyclic dependencies before producing a DAG.
- Recovering from a failed task requires recompiling from the original intent, not mutating the DAG.
- DAG storage is append-only. Historical DAGs remain available for replay and audit.
